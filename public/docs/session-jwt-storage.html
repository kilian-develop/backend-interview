인증 방식 · 저장소 · 보안 · 면접 필수

세션 vs JWT
쿠키 vs 스토리지

인증 상태를 서버가 관리하는가(세션) vs 토큰으로 자증하는가(JWT), 그리고 토큰을 어디에 저장하고 어떻게 전달하는가의 모든 트레이드오프

PART 1 — 세션 vs JWT
인증 상태를 어디서 관리하는가의 차이. 서버가 기억하는가(Stateful) vs 토큰이 자증하는가(Stateless)
🗄️ SESSION
1
로그인 요청
ID/PW를 서버로 전송
POST /login {id: "kilian", pw: "***"}
2
서버 — 세션 생성 및 저장
서버 메모리 / Redis에 세션 데이터 저장
# 서버 세션 저장소 (Redis) sessions["sid_abc123"] = { userId: 42, role: "admin", createdAt: "2024-01-15" }
3
Session ID만 쿠키로 전달
실제 데이터는 서버에, ID만 클라이언트로
Set-Cookie: JSESSIONID=sid_abc123; HttpOnly; Secure; SameSite=Lax
4
이후 요청마다 세션 조회
서버가 저장소를 조회해 사용자 확인
Cookie: JSESSIONID=sid_abc123 # 서버: 저장소 조회 → userId 확인
5
로그아웃 — 즉시 무효화
서버에서 세션 삭제 → 쿠키가 남아있어도 인증 실패
# sessions["sid_abc123"] 삭제 → 즉시 로그아웃 완료
🔐 JWT
1
로그인 요청
ID/PW를 서버로 전송
POST /login {id: "kilian", pw: "***"}
2
서버 — JWT 생성 (저장소 없음)
사용자 정보를 토큰에 담고 비밀키로 서명
# 서버 저장소 불필요! payload = {sub:42, role:"admin", exp:1700003600} token = sign(payload, SECRET_KEY)
3
AccessToken + RefreshToken 발급
두 토큰을 클라이언트에 전달
{ accessToken: "eyJ...짧은만료(15분)", refreshToken: "eyJ...긴만료(7일)" }
4
이후 요청 — 서명 검증만
저장소 조회 없이 서명과 만료시간만 검증
Authorization: Bearer eyJhbGci... # 서버: 서명 검증 → payload 추출 # DB 조회 없음 → 빠름
5
로그아웃 — 클라이언트에서 토큰 삭제
서버는 토큰을 기억 못함 → 만료 전까지 유효
# 클라이언트: 토큰 삭제 ⚠ 탈취된 토큰은 만료까지 유효 # 해결: 짧은 만료 + 블랙리스트 DB
항목 🗄️ 세션 (Session) 🔐 JWT

실전 권장 패턴
세션과 JWT 각각의 실무 최적 구성
🗄️ 세션 방식 권장 구성
# 1. 로그인 성공 SESSION_ID 생성 → Redis에 저장 # 2. 클라이언트 전달 Set-Cookie: JSESSIONID=sid_abc; HttpOnly; Secure; SameSite=Lax # 3. 다중 서버 확장 시 모든 서버 → 공유 Redis 클러스터 연결 # 4. 로그아웃 Redis에서 sid_abc 삭제 → 즉시 무효화
🔐 JWT 권장 구성 (Rotation 포함)
# 1. AccessToken → 메모리 저장 window.__accessToken = "eyJ...15분 만료" # 2. RefreshToken → HttpOnly 쿠키 Set-Cookie: refreshToken=eyJ...; HttpOnly; Secure; SameSite=Strict # 3. 만료 시 자동 갱신 (Rotation) POST /auth/refresh (RefreshToken 전송) → 기존 RefreshToken 무효화 → 새 AccessToken + 새 RefreshToken 발급 # 4. RefreshToken 재사용 감지 시 → 해당 유저 전체 로그아웃 (탈취 감지)
🎯 면접 핵심 포인트
🔄
세션이 다중 서버에서 문제가 되는 이유
서버 A에서 로그인해 세션을 만들었는데, 다음 요청이 서버 B로 가면 세션이 없어서 인증 실패합니다. Redis 같은 공유 세션 저장소로 해결합니다.
⏱️
JWT 로그아웃이 어려운 이유
서버가 발급한 JWT를 저장하지 않아서 취소할 방법이 없습니다. 만료(exp)까지 유효합니다. 해결책은 ① AccessToken 만료를 짧게 ② 블랙리스트 DB 운용입니다.
🍪
쿠키 HttpOnly가 왜 중요한가
HttpOnly 없으면 JS에서 document.cookie로 접근 가능합니다. XSS 공격 시 쿠키에 담긴 세션ID·토큰이 즉시 탈취됩니다.
🚫
localStorage에 토큰 저장이 위험한 이유
JS로 자유롭게 접근 가능해서 XSS 공격 한 번으로 모든 사용자의 토큰을 탈취할 수 있습니다. AccessToken은 메모리에, RefreshToken은 HttpOnly 쿠키에 저장합니다.
🛡️
SameSite 쿠키 속성이 해결하는 것
외부 사이트에서 유발된 요청에 쿠키를 자동 전송하지 않게 막아 CSRF를 방어합니다. Strict(완전 차단) / Lax(GET만 허용) / None(항상 전송).
🔁
Refresh Token Rotation이란
RefreshToken을 사용할 때마다 새 RefreshToken으로 교체합니다. 이전 RefreshToken을 재사용하면 탈취로 간주하고 해당 유저 전체 세션을 강제 로그아웃합니다.
