TLS가 사용하는 두 가지 암호화 방식
🔑
비대칭 암호화 (Asymmetric)
공개키(Public Key)와 개인키(Private Key) 쌍을 사용합니다. 공개키로 암호화한 데이터는 개인키로만 복호화할 수 있습니다.

TLS Handshake 과정에서 세션 키를 안전하게 교환할 때 사용합니다. 연산 비용이 높아 전체 통신에 쓰기엔 비효율적입니다.
알고리즘 : RSA, ECC, Diffie-Hellman
사용 시점: Handshake (키 교환 단계)
🗝️
대칭 암호화 (Symmetric)
암호화와 복호화에 동일한 하나의 세션 키를 사용합니다. 연산 속도가 빨라 실제 데이터 전송에 사용됩니다.

TLS Handshake를 통해 교환된 키로 이후 모든 HTTP 데이터를 암호화합니다.
알고리즘 : AES-256, ChaCha20
사용 시점: 실제 데이터 전송 단계
💡 왜 두 방식을 혼합해서 쓸까?
비대칭 암호화는 키를 안전하게 전달할 수 있지만 느립니다. 대칭 암호화는 빠르지만 키를 안전하게 전달하기 어렵습니다. TLS는 비대칭 암호화로 대칭키를 교환하고, 이후 대칭키로 실제 통신을 암호화하여 둘의 장점을 모두 취합니다.

HTTP의 취약점 — MITM 공격
HTTP는 데이터를 평문(plain text)으로 전송합니다. 공격자가 네트워크 중간에서 패킷을 가로채면 내용을 그대로 읽거나 조작할 수 있습니다.

중간자 공격(Man-In-The-Middle, MITM)은 공격자가 통신 양쪽 사이에 끼어들어 패킷을 가로채고 수정합니다. HTTPS는 ① 암호화로 내용 열람을 막고, ② 인증서로 서버 신원을 검증하여 이 공격을 방어합니다.

TLS의 3가지 목적
① 기밀성 — 암호화로 내용 열람 방지
② 무결성 — MAC으로 위변조 감지
③ 인증 — 인증서로 서버 신원 확인
⚡
TLS 1.3의 성능 개선
TLS 1.2는 2 RTT, TLS 1.3은 1 RTT로 Handshake 단계 축소. 재연결 시 0-RTT도 지원해 HTTPS 성능 우려가 사실상 해소됨
🏅
CA와 인증서 체인
Root CA → Intermediate CA → 서버 인증서 순으로 신뢰를 연결. 브라우저는 Root CA까지 체인을 검증하여 신뢰 여부를 판단
🌐
HTTP/2, HTTP/3
HTTP/2는 TLS가 사실상 필수, HTTP/3(QUIC)은 TLS 1.3 내장. 최신 프로토콜은 HTTPS 위에서만 제대로 동작
