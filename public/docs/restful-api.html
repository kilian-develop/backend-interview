RESTful API

REST의 6가지 제약 조건, HTTP 메서드와 상태 코드의 올바른 사용, URI 설계 원칙까지 — 실전 면접에서 꺼낼 수 있는 완벽 가이드

REST란 무엇인가?

REST (Representational State Transfer)는 2000년 Roy Fielding의 박사 논문에서 제안된 웹 아키텍처 스타일입니다.

REST는 특정 기술이 아닌 6가지 제약 조건(Constraints)의 집합이며, 이 조건을 지킨 API를 RESTful API라고 부릅니다.

HTTP의 설계 철학을 그대로 활용하기 때문에 별도 프로토콜 없이 HTTP만으로 구현 가능합니다.

# REST의 핵심 구성 요소 Resource (자원) URI로 자원을 식별 /users, /posts/123, /orders Representation (표현) 자원의 상태를 JSON/XML로 표현 { "id": 1, "name": "Kilian" } State Transfer (상태 전달) HTTP 메서드로 자원의 상태를 변경 GET · POST · PUT · DELETE
REST의 6가지 제약 조건 핵심
Constraint 01
🖥️ 클라이언트-서버 구조
UI(클라이언트)와 데이터 저장(서버)을 분리합니다. 서로 독립적으로 발전 가능하며 클라이언트는 URI만 알면 됩니다.
Client ↔ Server 독립 배포 가능
Frontend / Backend 분리 구조
Constraint 02
🔒 무상태 (Stateless)
서버는 클라이언트의 상태를 저장하지 않습니다. 각 요청은 처리에 필요한 모든 정보를 담아야 합니다. 세션 대신 JWT 토큰이 사용되는 이유입니다.
Authorization: Bearer {JWT}
요청 자체가 완결성을 가져야 함
Constraint 03
💾 캐시 가능 (Cacheable)
응답은 캐시 가능 여부를 명시해야 합니다. GET 응답은 캐시되어 성능을 향상시킵니다. Cache-Control, ETag 헤더를 활용합니다.
Cache-Control: max-age=3600
ETag: "33a64df551"
Constraint 04
🌐 균일한 인터페이스 (Uniform Interface)
REST의 핵심. URI로 자원을 식별하고 HTTP 메서드로 행위를 표현합니다. 자원과 행위를 분리하는 것이 핵심입니다.
자원: /users/1
행위: GET(조회) DELETE(삭제)
Constraint 05
🔗 계층형 시스템 (Layered System)
클라이언트는 직접 서버에 연결됐는지, 중간 서버(로드밸런서, CDN, API Gateway)에 연결됐는지 알 수 없습니다. 각 계층은 독립적입니다.
Client → API GW → LB → Server
중간 계층이 투명하게 동작
Constraint 06 (선택)
📦 코드 온 디맨드 (Code on Demand)
선택적 제약. 서버가 클라이언트에게 실행 가능한 코드(JavaScript)를 전달할 수 있습니다. 유일하게 선택적인 조건입니다.
예: 서버에서 JS 코드를 응답으로 전달
(현대 웹에서는 잘 사용되지 않음)
HTTP 메서드 — 행위를 표현하는 방법
GET

자원 조회 (Read)

서버에서 자원을 가져옵니다. 서버 상태를 변경하지 않습니다.

✓ 안전 (Safe)✓ 멱등 (Idempotent)✓ 캐시 가능✗ 요청 Body 없음
# 목록 조회 GET /api/users Authorization: Bearer {token} # 단건 조회 GET /api/users/42 # 쿼리 파라미터 GET /api/users?page=1&limit=20
# 200 OK 응답 200 OK { "id": 42, "name": "Kilian", "email": "k@example.com" } # 없을 경우 404 Not Found
POST

자원 생성 (Create)

새로운 자원을 생성합니다. 요청마다 새 자원이 만들어지므로 멱등하지 않습니다.

✗ 비안전 (Unsafe)✗ 비멱등✗ 캐시 불가✓ 요청 Body 있음
# 새 유저 생성 POST /api/users Content-Type: application/json { "name": "Kilian", "email": "k@example.com" }
# 201 Created 응답 201 Created Location: /api/users/43 { "id": 43, "name": "Kilian", "createdAt": "2024-01-15" }
PUT

자원 전체 수정 (Replace)

자원 전체를 교체합니다. 같은 요청을 여러 번 보내도 결과가 같습니다(멱등).

✗ 비안전✓ 멱등 (Idempotent)✗ 캐시 불가✓ 요청 Body 있음
# 유저 전체 교체 (모든 필드 필요) PUT /api/users/42 { "name": "Kilian Kim", "email": "new@example.com", "role": "admin" } # 누락 필드는 null이 됨!
# 200 OK 응답 200 OK { "id": 42, "name": "Kilian Kim", "email": "new@example.com", "role": "admin" }
PATCH

자원 부분 수정 (Partial Update)

자원의 일부만 수정합니다. PUT과 달리 변경할 필드만 전송합니다.

✗ 비안전△ 멱등 아닐 수 있음✗ 캐시 불가✓ 요청 Body 있음
# 이름만 수정 (부분 업데이트) PATCH /api/users/42 { "name": "Kilian Kim" } # email, role 등은 그대로 유지됨
# 200 OK 응답 200 OK { "id": 42, "name": "Kilian Kim", "email": "k@example.com" }
DELETE

자원 삭제 (Delete)

자원을 삭제합니다. 같은 요청을 반복해도 결과가 같습니다(멱등).

✗ 비안전✓ 멱등 (Idempotent)✗ 캐시 불가✗ 요청 Body 없음
# 유저 삭제 DELETE /api/users/42 Authorization: Bearer {token} # 재요청해도 멱등 DELETE /api/users/42 # 두 번째: 404 Not Found
# 성공: 204 No Content 204 No Content # 이미 삭제된 경우 404 Not Found # 권한 없는 경우 403 Forbidden

HTTP 상태 코드 — 응답을 명확하게
2xx 성공
200

OK

요청 성공. GET, PUT, PATCH에 사용

201

Created

자원 생성 성공. POST 후 Location 헤더와 함께

204

No Content

성공하였으나 응답 Body 없음. DELETE에 사용

206

Partial Content

Range 요청의 부분 응답. 스트리밍에 사용

3xx 리다이렉션
301

Moved Permanently

URI가 영구 변경됨. 새 URI로 리다이렉트

304

Not Modified

캐시 유효. 데이터 재전송 불필요 (ETag 활용)

4xx 클라이언트 오류
400

Bad Request

잘못된 요청 형식 또는 유효성 검사 실패

401

Unauthorized

인증 필요 — 로그인하지 않은 상태

403

Forbidden

인증됐지만 권한 없음 (401과 구분 중요!)

404

Not Found

자원이 존재하지 않음

405

Method Not Allowed

해당 URI에서 지원하지 않는 메서드

409

Conflict

중복 데이터 등 충돌. 이메일 중복 등에 사용

422

Unprocessable Entity

형식은 맞지만 비즈니스 유효성 실패

429

Too Many Requests

Rate Limit 초과. API 요청 제한에 사용

5xx 서버 오류
500

Internal Server Error

서버 내부 오류. 가능하면 구체적인 코드 사용 권장

502

Bad Gateway

게이트웨이가 업스트림에서 잘못된 응답 수신

503

Service Unavailable

서버 과부하 또는 점검 중. Retry-After 헤더 활용

메서드안전 (Safe)멱등 (Idempotent)캐시 가능설명

POST✗✗✗매번 새 자원 생성 → 중복 생성 주의

🎯 면접 핵심 포인트
🔄
Stateless가 왜 중요한가?
서버가 클라이언트 상태를 저장하지 않아 스케일 아웃이 쉽습니다. 어느 서버로 요청이 가도 동일하게 처리되어 로드밸런서 뒤에 서버를 수평 확장할 수 있습니다.
🔢
401 vs 403 차이
401 Unauthorized: 인증되지 않음 (로그인 필요). 403 Forbidden: 인증은 됐지만 권한이 없음. 면접에서 자주 헷갈리는 포인트입니다.
♻️
PUT vs PATCH
PUT은 자원 전체를 교체합니다. 누락 필드는 null이 됩니다. PATCH는 보낸 필드만 수정합니다. 실무에서는 PATCH를 더 많이 사용합니다.
🌐
REST vs RESTful
REST는 아키텍처 스타일(제약 조건의 집합)이고, RESTful은 그 조건을 충족한 구현을 말합니다. 많은 API가 REST스럽지만 완전한 RESTful은 아닙니다 (HATEOAS 미준수 등).
🔗
HATEOAS란?
Hypermedia As The Engine Of Application State. 응답에 관련 링크를 포함시켜 클라이언트가 다음 액션을 서버가 안내합니다. 완전한 REST의 조건이지만 실무에서 잘 구현되지 않습니다.
🆚
REST vs GraphQL
REST는 자원 중심 고정 엔드포인트. GraphQL은 단일 엔드포인트에서 원하는 데이터만 쿼리로 요청합니다. Over-fetching/Under-fetching 문제를 GraphQL이 해결하지만, REST가 단순하고 캐싱이 유리합니다.
