01
URL 파싱 & HTTP 요청 메시지 생성
L7 응용

브라우저는 주소창에 입력된 URL을 scheme, host, port, path, query, fragment로 분해합니다. 이 정보를 바탕으로 HTTP 요청 메시지(Request Line + Headers + Body)를 구성하고, OS의 소켓 라이브러리에 송신을 의뢰합니다.

▼ 자세히 보기
# URL 구조 분해 scheme : https ← 프로토콜 (HTTP/HTTPS/FTP 등) host : www.example.com ← 도메인 (DNS 조회 대상) port : 443 ← HTTPS 기본 포트 (생략 시 443) path : /api/data ← 리소스 경로 query : ?page=1&size=10 ← 쿼리 파라미터 fragment : #section ← 서버에 전송되지 않음 (클라이언트 전용) # 브라우저가 생성하는 HTTP Request Message GET /api/data?page=1&size=10 HTTP/1.1 Host: www.example.com User-Agent: Mozilla/5.0 (Chrome/120.0) Accept: application/json Accept-Language: ko-KR,ko;q=0.9 Accept-Encoding: gzip, deflate, br Connection: keep-alive Cookie: session_id=abc123
HSTS 체크

브라우저는 HSTS(HTTP Strict Transport Security) 목록을 확인합니다. 목록에 있으면 사용자가 http://로 입력해도 자동으로 https://로 변환합니다. 307 Internal Redirect가 발생합니다.

브라우저 캐시 확인

요청 전에 먼저 브라우저의 HTTP 캐시를 확인합니다. Cache-Control, ETag, Last-Modified 헤더를 기반으로 캐시 유효성을 판단하며, 유효하면 네트워크 요청 없이 캐시를 사용합니다. (304 Not Modified)

소켓 라이브러리

브라우저는 직접 네트워크를 제어하지 않습니다. OS의 Socket API(connect, write, read)를 호출하여 프로토콜 스택에 송수신을 위임합니다. DNS 조회도 Socket 라이브러리의 gethostbyname() 함수로 수행합니다.

02
DNS Lookup — 도메인을 IP 주소로 변환
L7 응용

네트워크 통신에는 IP 주소가 필요합니다. 브라우저는 브라우저 캐시 → OS 캐시 → hosts 파일 → Recursive Resolver → Root → TLD → Authoritative DNS 순서로 조회하여 도메인에 대응하는 IP 주소를 획득합니다.

▼ 자세히 보기
# DNS 조회 순서 (캐시 히트 시 즉시 반환) 1단계 브라우저 DNS 캐시 → chrome://net-internals/#dns 2단계 OS DNS 캐시 → systemd-resolved (Linux) / dscacheutil (Mac) 3단계 /etc/hosts 파일 → 로컬 호스트 매핑 (127.0.0.1 localhost 등) 4단계 Recursive Resolver (ISP) → 8.8.8.8 (Google) / 1.1.1.1 (Cloudflare) 5단계 Root DNS (.) → 전 세계 13개 루트 서버 (a~m.root-servers.net) 6단계 TLD DNS (.com) → .com 도메인 관리 서버 7단계 Authoritative DNS → example.com의 실제 A 레코드 보유 # DNS 응답 www.example.com → 93.184.216.34 (A Record, TTL: 3600) # DNS는 보통 UDP 포트 53을 사용 (512바이트 초과 시 TCP)
재귀 vs 반복 쿼리

클라이언트 → Resolver는 재귀 쿼리(최종 답까지 대신 조회). Resolver → 각 DNS 서버는 반복 쿼리(다음 서버 주소만 알려줌). 두 방식을 혼합하여 효율적으로 동작합니다.

TTL과 캐싱 전략

DNS 응답에 포함된 TTL(Time To Live)만큼 각 계층에서 캐싱합니다. TTL이 길면 성능은 좋지만 IP 변경 시 전파가 느리고, 짧으면 항상 최신이지만 DNS 서버 부하가 증가합니다.

DNS 보안

일반 DNS는 평문이라 DNS Spoofing, Cache Poisoning 공격에 취약합니다. DNSSEC(디지털 서명), DoH(DNS over HTTPS), DoT(DNS over TLS)로 보안을 강화할 수 있습니다.

03
ARP — IP 주소를 MAC 주소로 변환
L3 네트워크 L2 데이터링크

실제 LAN 통신은 IP 주소가 아닌 MAC 주소(48비트 물리 주소)로 이루어집니다. ARP(Address Resolution Protocol)를 사용하여 목적지(또는 게이트웨이)의 IP에 대응하는 MAC 주소를 획득합니다.

▼ 자세히 보기
# ARP 동작 과정 1) OS가 목적지 IP(93.184.216.34)를 확인 2) 같은 서브넷인지 판단 (서브넷 마스크 & 연산) → 다른 네트워크이므로 기본 게이트웨이(라우터)의 MAC이 필요 3) ARP 캐시 테이블 확인 → 없으면 ARP Request 브로드캐스트 4) 게이트웨이가 ARP Reply로 자신의 MAC 주소 응답 # ARP Request (브로드캐스트) "IP 192.168.1.1의 MAC 주소를 알려주세요!" → LAN 내 모든 장비에게 전송 (FF:FF:FF:FF:FF:FF) # ARP Reply (유니캐스트) "192.168.1.1은 AA:BB:CC:DD:EE:01 입니다." → 요청자에게만 응답 # ARP 캐시 확인 명령 $ arp -a # macOS/Linux > arp -a # Windows
왜 라우터의 MAC이 필요할까?

목적지 서버가 다른 네트워크에 있으므로, Ethernet Frame의 목적지 MAC은 게이트웨이(라우터)의 MAC 주소가 됩니다. IP 헤더의 목적지 IP는 서버 IP(93.184.216.34)를 유지합니다. 라우터에서 hop마다 MAC은 변경되지만 IP는 유지됩니다.

ARP Spoofing 공격

ARP는 인증 없이 응답을 신뢰합니다. 공격자가 위조 ARP Reply를 보내 게이트웨이를 사칭하면 트래픽을 가로챌 수 있습니다. Dynamic ARP Inspection(DAI)과 802.1X 인증으로 방어합니다.

ARP vs NDP

ARP는 IPv4 전용입니다. IPv6에서는 NDP(Neighbor Discovery Protocol)가 같은 역할을 수행하며, ICMPv6 기반으로 더 안전하게 동작합니다. Solicited-Node Multicast를 사용하여 브로드캐스트 부하도 줄였습니다.

04
TCP 3-Way Handshake — 가상 회선 수립
L4 전송

OS의 프로토콜 스택이 소켓을 생성(socket)하고, 서버의 443번 포트로 TCP 3-Way Handshake를 수행합니다. SYN → SYN+ACK → ACK 교환으로 양방향 통신 경로를 확인하고, 시퀀스 번호를 동기화합니다.

▼ 자세히 보기
# TCP 3-Way Handshake — 시퀀스 번호 교환 Step 1 Client → Server : SYN seq=x(임의 ISN) └─ 클라이언트 상태: CLOSED → SYN_SENT Step 2 Server → Client : SYN+ACK seq=y(서버 ISN), ack=x+1 └─ 서버 상태: LISTEN → SYN_RCVD Step 3 Client → Server : ACK ack=y+1 └─ 양쪽 상태: ESTABLISHED ✅ # 왜 ISN은 랜덤 난수? # 1) 이전 연결의 지연 패킷과 현재 연결의 패킷을 구분 # 2) TCP 시퀀스 번호 예측 공격 방지 (보안) # 소켓 API 호출 순서 Client: socket() → connect() ← 여기서 3-way 수행 Server: socket() → bind() → listen() → accept()
왜 3-way인가? (2-way가 아닌 이유)

2-way로는 클라이언트의 수신 능력을 서버가 확인할 수 없습니다. 3번째 ACK으로 클라이언트가 서버의 SYN+ACK를 정상 수신했음을 확인합니다. 또한 지연 도착한 과거 SYN이 새 연결로 오인되는 것을 방지합니다.

TCP 옵션 협상

Handshake 중에 MSS(Maximum Segment Size), Window Scale, Selective ACK(SACK), Timestamp 등의 옵션을 협상합니다. 이 값들이 이후 데이터 전송의 효율성과 신뢰성을 결정합니다.

SYN Flooding 공격

공격자가 위조 IP로 대량의 SYN을 보내면 서버의 반개방(half-open) 큐가 가득 찹니다. SYN Cookie: 큐를 사용하지 않고 SYN+ACK의 시퀀스 번호에 연결 정보를 인코딩하여 ACK 수신 시 검증하는 방어 기법입니다.

05
TLS Handshake — 암호화 통신 협상 (HTTPS)
L6 표현 L5 세션

HTTPS이므로 TCP 연결 위에 TLS Handshake를 수행합니다. 서버 인증(인증서 검증), 암호화 알고리즘 합의, 대칭키 교환을 거쳐 이후 모든 HTTP 통신을 암호화합니다. TLS 1.3 기준 1-RTT로 완료됩니다.

▼ 자세히 보기
# TLS 1.3 Handshake (1-RTT) 1) Client → Server : ClientHello - 지원 TLS 버전: 1.3 - 지원 Cipher Suite: TLS_AES_256_GCM_SHA384 등 - Key Share: ECDHE 공개값 (키 교환 파라미터) - Random: 32바이트 난수 2) Server → Client : ServerHello + Certificate + Finished - 선택한 Cipher Suite - Key Share: 서버 ECDHE 공개값 - Certificate: 서버 인증서 (CA 서명 포함) - Certificate Verify: 개인키로 서명한 핸드셰이크 해시 - Finished: 핸드셰이크 완료 확인 MAC 3) Client → Server : Finished - 인증서 체인 검증 (Root CA → Intermediate CA → 서버 인증서) - ECDHE로 Pre-Master Secret 계산 → Master Secret 유도 - 양쪽 모두 동일한 대칭키(Session Key) 생성 완료 ✅ # 이후 모든 HTTP 데이터는 대칭키(AES-256-GCM)로 암호화 # TLS 1.2는 2-RTT 필요, TLS 1.3은 1-RTT (재연결 시 0-RTT 가능)
하이브리드 암호화

비대칭키(ECDHE)로 안전하게 키를 교환하고, 대칭키(AES)로 실제 데이터를 암호화합니다. 비대칭키는 느리지만 안전한 키 교환이 가능하고, 대칭키는 빠릅니다. 양쪽의 장점을 조합한 것입니다.

전방향 비밀성 (Forward Secrecy)

ECDHE는 매 세션마다 새로운 임시 키를 생성합니다. 서버의 개인키가 나중에 유출되더라도 과거에 주고받은 데이터는 복호화할 수 없습니다. RSA 키 교환은 이 속성이 없어 TLS 1.3에서 제거되었습니다.

인증서 신뢰 체인

서버 인증서 → 중간 CA → Root CA 순으로 서명을 검증합니다. 브라우저는 신뢰할 수 있는 Root CA 목록을 내장하고 있어서, 체인이 끊어지거나 Root CA를 신뢰할 수 없으면 경고를 표시합니다.

06
HTTP 요청 전송 — 패킷 캡슐화
L7 응용 L4 전송 L3 네트워크 L2 데이터링크

HTTP 요청 메시지가 각 계층을 거치며 헤더가 추가(캡슐화)됩니다. TLS로 암호화된 HTTP 메시지에 TCP 헤더(포트, 시퀀스) → IP 헤더(출발/목적지 IP) → Ethernet 프레임(MAC 주소) → 전기신호로 변환되어 LAN 어댑터(NIC)에서 송출됩니다.

▼ 자세히 보기
# 캡슐화 과정 (위 → 아래) L7 HTTP 메시지 (TLS 암호화 완료) │ GET /api/data HTTP/1.1 ... ▼ L4 + TCP Header (20 bytes) │ SRC Port: 54321 | DST Port: 443 │ Seq: x+1 | Ack: y+1 | Window: 65535 │ Flags: PSH, ACK ▼ L3 + IP Header (20 bytes) │ SRC IP: 192.168.1.10 │ DST IP: 93.184.216.34 │ TTL: 64 | Protocol: TCP(6) ▼ L2 + Ethernet Frame (14 bytes header + 4 bytes FCS) │ SRC MAC: AA:BB:CC:11:22:33 (내 NIC) │ DST MAC: AA:BB:CC:DD:EE:01 (게이트웨이) │ EtherType: 0x0800 (IPv4) ▼ L1 전기신호/광신호로 변환 → NIC에서 송출
MSS와 단편화(Fragmentation)

HTTP 메시지가 MSS(보통 1460 bytes)보다 크면 TCP가 여러 세그먼트로 분할합니다. 각 세그먼트에 시퀀스 번호가 부여되어 수신측에서 원래 순서로 재조합합니다. IP 레벨에서도 MTU(1500 bytes) 초과 시 단편화가 발생할 수 있습니다.

NAT (Network Address Translation)

가정/회사 네트워크는 사설 IP(192.168.x.x)를 사용합니다. 라우터의 NAT가 사설 IP → 공인 IP로 변환합니다. 출발지 IP/Port를 라우터의 공인 IP/임시 Port로 교체하고, 응답이 오면 다시 변환합니다.

07
LAN → ISP → 인터넷 — 패킷 라우팅
L3 네트워크 L2 데이터링크 L1 물리

패킷이 라우터 → ISP → IX(인터넷 교환점) → 목적지 ISP → 목적지 LAN을 거쳐 이동합니다. 각 라우터에서 라우팅 테이블을 참조하여 다음 홉(Next Hop)을 결정하며, 홉을 지날 때마다 Ethernet Frame의 MAC 주소가 교체됩니다.

▼ 자세히 보기
# 패킷 이동 (Hop by Hop) [내 PC] 192.168.1.10 │ L2: 내 MAC → 라우터 MAC ▼ [가정용 라우터] 192.168.1.1 / 공인IP: 203.0.113.5 │ NAT: 사설IP → 공인IP 변환 │ L2: 라우터 MAC → ISP 장비 MAC ▼ [ISP 엣지 라우터] │ BGP/OSPF로 최적 경로 결정 ▼ [인터넷 핵심부] 고속 라우터들 (여러 홉) │ 각 홉에서: IP 헤더의 TTL 값 -1 │ 새로운 MAC 주소로 Frame 교체 ▼ [목적지 ISP 라우터] ▼ [목적지 LAN] 93.184.216.34 # 경로 추적 $ traceroute www.example.com 1 192.168.1.1 ← 게이트웨이 2 10.0.0.1 ← ISP 내부 3 72.14.236.68 ← 인터넷 핵심부 ... 12 93.184.216.34 ← 목적지 도달!
IP는 유지, MAC은 변경

패킷이 라우터를 거칠 때마다 IP 헤더(L3)는 동일하게 유지되지만, Ethernet Frame(L2)의 MAC 주소는 매 홉마다 변경됩니다. IP는 최종 목적지, MAC은 다음 홉까지의 전달을 담당합니다.

TTL (Time To Live)

IP 헤더의 TTL은 라우터를 하나 지날 때마다 1씩 감소합니다. 0이 되면 패킷이 폐기되고 ICMP Time Exceeded 메시지가 발신자에게 반환됩니다. 이를 이용한 것이 traceroute 명령입니다.

BGP (Border Gateway Protocol)

인터넷의 자율 시스템(AS) 간 경로 교환 프로토콜입니다. 각 ISP/기업은 AS 번호를 가지며, BGP가 AS 간 최적 경로를 결정합니다. 인터넷의 "우편번호 시스템"이라 할 수 있습니다.

08
서버 측 처리 — 방화벽, 로드밸런서, 웹 서버
L7 응용 L4 전송

서버 LAN에 도착한 패킷은 방화벽 → CDN/캐시 → 로드밸런서 → Reverse Proxy → 웹 서버(WAS)를 거칩니다. 웹 서버는 역캡슐화로 HTTP 메시지를 복원하고, 라우팅 → 비즈니스 로직 → DB 조회 → HTTP 응답을 생성합니다.

▼ 자세히 보기

방화벽 (L4/L7)

L4 방화벽: IP/포트 기반 필터링 (TCP 443만 허용). L7 방화벽(WAF): HTTP 내용까지 검사하여 SQL Injection, XSS 등 악성 요청을 차단합니다. AWS WAF, Cloudflare WAF 등이 대표적입니다.

로드 밸런서

L4 LB: IP/포트 기반으로 빠르게 분산 (NLB). L7 LB: URL 경로, 헤더 등으로 세밀한 라우팅 (ALB). Round Robin, Least Connection, IP Hash 등의 알고리즘으로 트래픽을 분산합니다.

Reverse Proxy

Nginx, HAProxy 등이 클라이언트 요청을 받아 내부 WAS에 전달합니다. SSL Termination(TLS 복호화를 프록시에서 처리), 정적 파일 서빙, 요청 캐싱, 압축(gzip/brotli) 등을 담당합니다.

09
브라우저 렌더링 파이프라인
L7 응용

서버의 응답(HTML/CSS/JS)이 브라우저에 도착하면, HTML Parsing → DOM Tree → CSSOM → Render Tree → Layout → Paint → Composite 순서로 화면을 렌더링합니다. 이 과정에서 추가 리소스(CSS, JS, 이미지)에 대한 요청이 병렬로 발생합니다.

▼ 자세히 보기
# 브라우저 렌더링 파이프라인 1) HTML Parsing → DOM Tree 구성 - HTML 토큰화 → 노드 생성 → 트리 구조 - <script> 태그 만나면 파싱 중단 (render-blocking) - defer/async 속성으로 중단 방지 가능 2) CSS Parsing → CSSOM(CSS Object Model) 구성 - 외부 CSS 파일 병렬 다운로드 - 선택자 매칭, 우선순위 계산 (specificity) - CSS도 render-blocking 리소스 3) Render Tree = DOM + CSSOM 결합 - 화면에 보이는 노드만 포함 (display:none 제외) - 각 노드에 계산된 스타일 적용 4) Layout (Reflow) - 각 노드의 정확한 위치와 크기 계산 - 뷰포트 기반 상대 단위(%, vh, vw) → 픽셀 변환 5) Paint (Repaint) - 레이어별 픽셀 채우기 (배경, 텍스트, 테두리, 그림자) - 복잡한 요소(box-shadow, gradient)는 별도 레이어 6) Composite - GPU가 레이어들을 합성하여 최종 화면 출력 - transform, opacity 변경은 Composite만 발생 (가장 빠름)
Reflow vs Repaint vs Composite

Reflow(Layout): width, height, margin 등 레이아웃 변경 시 발생 (비용 최대). Repaint: color, background 등 시각적 변경 시 발생. Composite Only: transform, opacity만 변경하면 Layout/Paint 없이 GPU 합성만 수행 (비용 최소).

Critical Rendering Path 최적화

CSS는 <head>에 배치(빠른 CSSOM 생성), JS는 <body> 하단 또는 defer/async 사용, 필수 CSS를 인라인으로 삽입(Critical CSS), 이미지는 lazy loading. First Contentful Paint(FCP)를 최소화하는 것이 목표입니다.

추가 리소스 요청

HTML 파싱 중 <link>, <script>, <img> 등을 만나면 새로운 HTTP 요청이 발생합니다. HTTP/2 멀티플렉싱으로 하나의 TCP 연결에서 병렬 요청이 가능하며, 브라우저의 preload scanner가 미리 필요한 리소스를 예측하여 요청합니다.

면접에서 자주 나오는 질문

브라우저에 URL을 입력하면 무슨 일이 일어나나요? (30초 요약)
URL 파싱 후 HTTP 요청 메시지를 생성합니다. DNS로 도메인을 IP로 변환하고, ARP로 게이트웨이 MAC 주소를 획득합니다. TCP 3-way Handshake로 연결을 수립하고, HTTPS면 TLS Handshake로 암호화를 협상합니다. HTTP 요청이 캡슐화(TCP→IP→Ethernet→전기신호)되어 라우터를 거쳐 서버에 도달합니다. 서버가 역캡슐화 후 요청을 처리하여 응답을 반환하면, 브라우저가 HTML Parsing → DOM/CSSOM → Render Tree → Layout → Paint → Composite 순서로 화면을 렌더링합니다.
TCP와 UDP의 차이점, 그리고 DNS가 UDP를 사용하는 이유는?
TCP는 연결 지향(3-way handshake), 순서/신뢰성 보장, 흐름/혼잡 제어를 제공하지만 오버헤드가 큽니다. UDP는 비연결형으로 신뢰성은 없지만 빠릅니다. DNS는 대부분 512바이트 이내의 작은 패킷이므로 TCP의 연결 수립 오버헤드가 불필요하고, 빠른 응답이 중요하기 때문에 UDP를 사용합니다. 다만 응답이 512바이트를 초과하거나 Zone Transfer 시에는 TCP를 사용합니다.
라우터를 거칠 때 IP 주소와 MAC 주소는 어떻게 변하나요?
IP 주소(L3)는 출발지에서 목적지까지 변하지 않습니다(NAT 제외). 반면 MAC 주소(L2)는 홉마다 변경됩니다. 각 라우터에서 Ethernet Frame을 벗기고(역캡슐화), IP 헤더를 확인하여 다음 홉을 결정한 뒤, 새로운 목적지 MAC 주소로 Frame을 다시 만들어(재캡슐화) 전달합니다.
TLS Handshake 과정에서 비대칭키와 대칭키는 어떻게 사용되나요?
TLS Handshake에서 ECDHE(비대칭키) 알고리즘으로 양쪽이 안전하게 Pre-Master Secret을 합의합니다. 이로부터 Master Secret을 유도하고, 실제 데이터 암호화에 사용할 대칭키(Session Key)를 생성합니다. 이후 모든 HTTP 통신은 이 대칭키(AES-256-GCM 등)로 암호화됩니다. 비대칭키는 안전한 키 교환에만 사용되고, 빠른 대칭키로 실제 데이터를 처리하는 하이브리드 방식입니다.
브라우저 렌더링에서 Reflow가 성능에 미치는 영향과 최적화 방법은?
Reflow(Layout)는 요소의 위치/크기를 재계산하므로 비용이 가장 큽니다. width, height, margin 등의 변경이나 DOM 추가/제거 시 발생하며, 하위 노드까지 연쇄적으로 영향을 줍니다. 최적화: transform/opacity로 애니메이션 처리(Composite Only), DOM 읽기/쓰기 분리(강제 동기 레이아웃 방지), will-change로 별도 레이어 생성, requestAnimationFrame 사용, 가상 스크롤링으로 DOM 노드 수 제한 등이 있습니다.
Connection: keep-alive와 HTTP/2 멀티플렉싱의 차이는?
HTTP/1.1의 Keep-Alive는 하나의 TCP 연결을 재사용하지만, 요청-응답이 순차적(FIFO)이므로 앞선 요청이 느리면 뒤의 요청도 대기합니다(HOL Blocking). HTTP/2의 멀티플렉싱은 하나의 TCP 연결에서 여러 스트림을 동시에 병렬 처리하여 HTTP 레벨의 HOL Blocking을 해결합니다. 다만 TCP 레벨 HOL Blocking은 남아있어, HTTP/3(QUIC/UDP)에서 근본적으로 해결되었습니다.

면접 핵심 포인트 정리 (9단계)

01
URL 파싱(scheme/host/path) → HTTP Request Message 생성 → HSTS 체크 → 브라우저 캐시 확인
02
DNS: 브라우저 캐시 → OS 캐시 → Resolver → Root → TLD → Authoritative (UDP 53)
03
ARP: 게이트웨이 IP → MAC 주소 변환 (브로드캐스트 → 유니캐스트 응답)
04
TCP 3-Way Handshake: SYN(ISN) → SYN+ACK → ACK (양방향 통신 확인 + 시퀀스 동기화)
05
TLS Handshake: ClientHello → ServerHello+Certificate → ECDHE 키 교환 → 대칭키 생성 (1-RTT)
06
캡슐화: HTTP(TLS) → TCP Header → IP Header → Ethernet Frame → 전기신호 (L7→L1)
07
라우팅: NAT 변환 → 라우터 hop-by-hop (IP 유지, MAC 변경) → ISP → 인터넷 → 목적지
08
서버: 방화벽/WAF → CDN → 로드밸런서 → Reverse Proxy → WAS → 역캡슐화 → 요청 처리 → 응답
09
렌더링: HTML Parsing → DOM + CSSOM → Render Tree → Layout → Paint → Composite
