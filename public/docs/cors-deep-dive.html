Section 01
SOP와 CORS — 왜 막히는가?
CORS를 이해하려면 먼저 브라우저의 기본 보안 정책인 SOP(Same-Origin Policy)를 알아야 합니다
🔒 SOP (Same-Origin Policy)
브라우저의 기본 보안 정책입니다. 다른 Origin의 리소스를 JS로 읽는 것을 기본적으로 차단합니다.

이것이 없으면 악성 사이트가 피해자의 브라우저를 이용해 은행 API를 몰래 호출하고 응답을 읽어갈 수 있습니다.
🔓 CORS (Cross-Origin Resource Sharing)
SOP의 예외를 허용하는 HTTP 헤더 기반 메커니즘입니다.

서버가 Access-Control-Allow-Origin 헤더로 "이 Origin은 내 리소스에 접근해도 돼"라고 명시적으로 허용합니다.
Origin = Scheme + Host + Port — 셋 중 하나라도 다르면 Cross-Origin
https://api.example.com:443/users
Scheme (프로토콜)
https
http vs https는 다른 Origin
반드시 일치해야 함
Host (도메인)
api.example.com
서브도메인도 다른 Origin
example.com ≠ api.example.com
Port
443
포트가 다르면 다른 Origin
:3000 ≠ :8080
✅ Same-Origin (허용)
# 기준: https://example.com:443 ✓ https://example.com/api path만 다름 ✓ https://example.com:443/data 동일
❌ Cross-Origin (기본 차단)
# 기준: https://example.com:443 ✗ http://example.com scheme 다름 ✗ https://api.example.com host 다름 ✗ https://example.com:8080 port 다름 ✗ https://other.com domain 다름
Section 02
CORS 요청 흐름 — Simple vs Preflight vs Credentialed
모든 Cross-Origin 요청이 Preflight를 발생시키는 것은 아닙니다. 요청 조건에 따라 세 가지로 나뉩니다
Simple Request 조건 (모두 만족해야 함)
메서드: GET POST HEAD 중 하나  |  Content-Type: application/x-www-form-urlencoded, multipart/form-data, text/plain 중 하나  |  커스텀 헤더 없음  →  Preflight 없이 바로 요청
BROWSER
GET /api/data
Origin: https://app.com
SERVER
── 브라우저 → 서버 ────────────────────────────── GET /api/data HTTP/1.1 Origin: https://app.com ── 서버 → 브라우저 ────────────────────────────── HTTP/1.1 200 OK Access-Control-Allow-Origin: https://app.com # 브라우저: Origin 일치 확인 → JS에서 응답 읽기 허용 ── 서버가 CORS 헤더를 안 보내면 ─────────────────── CORS Error: No 'Access-Control-Allow-Origin' header # 요청은 서버에 도달했지만, 브라우저가 응답을 JS에 전달 안 함
⚠️ 중요: CORS 오류는 서버가 요청을 거부한 게 아닙니다. 요청은 서버에 도달했고 응답도 왔습니다. 브라우저가 CORS 헤더를 확인하고 JS에게 응답을 숨긴 것입니다.
Preflight가 발생하는 조건 (하나라도 해당되면)
메서드: PUT DELETE PATCH OPTIONS 등  |  Content-Type: application/json 등  |  커스텀 헤더 포함 (Authorization 등)  →  OPTIONS 사전 요청을 먼저 보냄
── STEP 1: 브라우저가 OPTIONS Preflight 자동 발송 ───── OPTIONS /api/data HTTP/1.1 Origin: https://app.com Access-Control-Request-Method: POST Access-Control-Request-Headers: Content-Type, Authorization # "POST 요청에 이런 헤더를 쓰려는데 허용하니?" ── STEP 2: 서버 → Preflight 응답 ────────────────────── HTTP/1.1 204 No Content Access-Control-Allow-Origin: https://app.com Access-Control-Allow-Methods: GET, POST, PUT, DELETE Access-Control-Allow-Headers: Content-Type, Authorization Access-Control-Max-Age: 86400 ← 이 결과를 24시간 캐싱 ── STEP 3: 허가 받은 후 실제 요청 전송 ──────────────── POST /api/data HTTP/1.1 Origin: https://app.com Content-Type: application/json Authorization: Bearer eyJ... Body: {"key": "value"} → 200 OK (실제 응답) ── Preflight 실패 시 ─────────────────────────────────── Preflight 응답에 해당 헤더 없음 → 실제 요청 전송 안 함
💡 Access-Control-Max-Age를 설정하면 같은 요청 조합에 대해 Preflight 결과를 캐싱합니다. 매 API 요청마다 OPTIONS가 날아가지 않아 성능이 개선됩니다.
Credentialed Request — 쿠키, Authorization 헤더, TLS 인증서 등 인증 정보를 포함한 Cross-Origin 요청.
기본적으로 Cross-Origin 요청에는 쿠키가 포함되지 않습니다. 명시적으로 설정해야 합니다.
── 클라이언트: credentials 옵션 필수 ───────────────── fetch('https://api.example.com/data', { method: 'GET', credentials: 'include' ← 이게 없으면 쿠키 안 감 }) ── XMLHttpRequest ───────────────────────────────────── xhr.withCredentials = true; ── 서버: 두 헤더 모두 필수 ─────────────────────────── Access-Control-Allow-Origin: https://app.com # ❌ * 와일드카드 불가! 정확한 Origin 명시 필수 Access-Control-Allow-Credentials: true ── 둘 중 하나라도 빠지면 ───────────────────────────── CORS Error: credentials mode 'include' 요청에 Access-Control-Allow-Origin이 '*' 이거나 Access-Control-Allow-Credentials가 없음
⚠️ 핵심 제약: Credentialed Request에서 Access-Control-Allow-Origin: * 와일드카드는 사용 불가입니다. 정확한 Origin을 명시해야 합니다.
Section 03
CORS 헤더 완전 레퍼런스
요청(브라우저)과 응답(서버) 헤더 각각의 역할과 사용법
📤 요청 헤더 (브라우저 자동 설정)
Origin
요청을 보내는 출처. 브라우저가 Cross-Origin 요청 시 자동으로 포함합니다.
Origin: https://app.example.com
Access-Control-Request-Method
Preflight에서 실제로 사용할 HTTP 메서드를 서버에 미리 알립니다.
Access-Control-Request-Method: POST
Access-Control-Request-Headers
Preflight에서 실제 요청에 포함할 커스텀 헤더 목록을 서버에 알립니다.
Access-Control-Request-Headers: Authorization, Content-Type
📥 응답 헤더 (서버 설정 필요)
Access-Control-Allow-Origin
접근을 허용할 Origin을 명시합니다. *는 와일드카드(단, Credentials 불가)
Access-Control-Allow-Origin: https://app.com
Access-Control-Allow-Methods
허용할 HTTP 메서드 목록입니다. Preflight 응답에 포함합니다.
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers
허용할 요청 헤더 목록입니다. Authorization 등 커스텀 헤더 포함 시 필수
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Allow-Credentials
쿠키·인증 정보 포함 허용 여부. true로 설정 시 Allow-Origin에 * 불가
Access-Control-Allow-Credentials: true
Access-Control-Max-Age
Preflight 결과를 브라우저가 캐싱할 시간(초). 반복 Preflight 방지
Access-Control-Max-Age: 86400
Access-Control-Expose-Headers
JS에서 읽을 수 있도록 노출할 응답 헤더 목록 (기본 6개 외 커스텀 헤더)
Access-Control-Expose-Headers: X-Total-Count
Section 04
CORS 해결 방법
환경별 올바른 해결 방법 — 개발 환경과 프로덕션을 구분해서 적용해야 합니다
1
서버에서 CORS 헤더 설정 — 프로덕션 권장
PROD
Spring Boot
// 방법 1: @CrossOrigin 어노테이션 @CrossOrigin(origins = "https://app.com") @GetMapping("/api/data") public ResponseEntity getData() {...} // 방법 2: WebMvcConfigurer (전역) @Configuration public class CorsConfig implements WebMvcConfigurer { @Override public void addCorsMappings(CorsRegistry r) { r.addMapping("/**") .allowedOrigins("https://app.com") .allowedMethods("GET","POST","PUT","DELETE") .allowedHeaders("*") .allowCredentials(true) .maxAge(3600); } }
Node.js / Express
// cors 패키지 사용 const cors = require('cors') app.use(cors({ origin: 'https://app.com', // 여러 Origin 허용 시 origin: ['https://app.com', 'https://m.app.com'], // 동적 허용 (DB 화이트리스트 등) origin: (origin, callback) => { if (allowList.includes(origin)) callback(null, true) else callback(new Error('Not allowed')) }, credentials: true, methods: ['GET','POST','PUT','DELETE'], maxAge: 86400 }))
2
프록시 서버 사용 — 개발 환경 / Nginx
DEV / PROD
Vite / CRA 개발 서버 프록시
// vite.config.ts export default { server: { proxy: { '/api': { target: 'http://localhost:8080', changeOrigin: true, // /api/users → /users 로 rewrite rewrite: (path) => path.replace(/^\/api/, '') } } } } // 브라우저: localhost:5173/api/users 요청 // Vite: localhost:8080/users 로 포워딩 // Same-Origin 이므로 CORS 없음!
Nginx 리버스 프록시 (프로덕션)
# nginx.conf server { listen 443 ssl; server_name app.com; # 프론트엔드 location / { root /usr/share/nginx/html; } # API 프록시 → Same-Origin 처리 location /api/ { proxy_pass http://backend:8080/; proxy_set_header Host $host; proxy_set_header X-Real-IP $remote_addr; } } # app.com/api/* → backend:8080/* # 브라우저 입장에서 모두 app.com → CORS 없음
Section 05
자주 하는 실수 Top 4
CORS 오류를 마주했을 때 흔히 잘못 접근하는 방법들
❌
* 와일드카드 + credentials: include 동시 사용
잘못된 방법
# 서버 응답 Access-Control-Allow-Origin: * Access-Control-Allow-Credentials: true # 클라이언트 fetch(url, { credentials: 'include' }) → CORS Error: wildcard + credentials 불가
올바른 방법
# 정확한 Origin을 명시해야 함 Access-Control-Allow-Origin: https://app.com Access-Control-Allow-Credentials: true # 여러 Origin은 요청의 Origin 헤더를 읽어 # 화이트리스트 확인 후 동적으로 설정
❌
OPTIONS Preflight 요청을 인증 미들웨어에서 차단
잘못된 방법
// JWT 인증 미들웨어가 OPTIONS도 막음 app.use((req, res, next) => { const token = req.headers.authorization if (!token) return res.status(401).send() // OPTIONS preflight에도 401 반환! next() })
올바른 방법
// CORS 미들웨어를 인증보다 먼저! app.use(cors(corsOptions)) ← 먼저 app.use((req, res, next) => { // OPTIONS는 인증 체크 건너뜀 if (req.method === 'OPTIONS') return next() // 나머지만 인증 검사 verifyToken(req, res, next) })
❌
CORS를 서버 보안 수단으로 착각
오해
# CORS가 막으면 서버가 안전하다? NO. CORS는 브라우저 정책입니다. # curl, Postman은 CORS를 무시 curl -H "Origin: https://evil.com" \ https://api.example.com/data # → 서버는 정상 응답 # 서버 입장에서는 차단한 게 아무것도 없음
올바른 이해
# CORS = 브라우저가 JS의 응답 읽기를 제한 # 실제 API 보안은 별도로 구현 필요 ✓ JWT / 세션 인증 ✓ API Key 검증 ✓ Rate Limiting ✓ Input validation # CORS는 UX 보호, 인증은 서버에서 따로!
❌
개발 편의로 모든 Origin 허용 (* ) 을 프로덕션에 그대로 배포
위험한 방법
# 개발 중 빠르게 해결하려다가... Access-Control-Allow-Origin: * # 프로덕션에도 그대로 배포 → 어떤 사이트에서도 API 접근 가능 → CSRF + XSS 공격 표면 확대
올바른 방법
# 환경별 Origin 화이트리스트 관리 const allowedOrigins = { development: ['http://localhost:3000'], production: ['https://app.com', 'https://m.app.com'] } origin: allowedOrigins[process.env.NODE_ENV]
Section 06
🎯 면접 핵심 포인트
🤔
CORS 오류는 서버가 거부한 것인가?
아닙니다. 요청은 서버에 도달했고 응답도 왔습니다. 브라우저가 CORS 헤더를 보고 JS에게 응답을 전달하지 않는 것입니다. curl·Postman에서 되는 이유가 바로 이것입니다.
✈️
Preflight는 왜 존재하는가?
부작용이 생길 수 있는 요청(PUT·DELETE·커스텀 헤더 등)을 실제로 보내기 전에 서버의 허가를 먼저 받기 위해서입니다. Simple Request는 부작용이 적어 바로 보냅니다.
🌐
SOP가 없으면 어떤 일이?
악성 사이트가 피해자 브라우저로 은행 API를 몰래 호출하고 응답(계좌 잔액 등)을 읽어갈 수 있습니다. SOP는 CSRF의 핵심 방어선 중 하나입니다.
⭐
* 와일드카드와 credentials
credentials: include와 Allow-Origin: *는 동시에 사용할 수 없습니다. 인증 정보 포함 시 반드시 정확한 Origin을 명시해야 합니다.
⚡
Preflight 성능 최적화
Access-Control-Max-Age로 Preflight 결과를 캐싱하면 매 요청마다 OPTIONS가 발생하지 않습니다. 브라우저마다 최대 캐싱 시간이 다릅니다 (Chrome: 2시간).
🔒
CORS vs CSRF 관계
CORS는 다른 Origin의 JS가 응답을 읽는 것을 막고, CSRF는 요청 자체를 위조하는 공격입니다. SameSite 쿠키와 함께 방어해야 합니다.
