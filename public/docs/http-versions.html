HTTP의 진화

1991년 단 하나의 메서드로 시작한 HTTP가 어떻게 멀티플렉싱, QUIC까지 발전했는지 — 버전별 핵심 차이와 각 버전이 해결한 문제들

HTTP/0.9
The One-Line Protocol
1991 · Tim Berners-Lee
GET Only HTML Only 헤더 없음
▼

HTTP의 원형. GET 메서드 하나와 요청 경로만 존재했습니다. 헤더도, 상태 코드도, 버전 표기도 없었습니다. 응답은 HTML 본문 그대로였고, 전송이 끝나면 TCP 연결이 즉시 닫혔습니다. 웹의 씨앗이라 할 수 있는 가장 단순한 형태입니다.

📄
단일 메서드 (GET)
GET만 존재. 요청 형식은 GET /index.html 한 줄이 전부였습니다.
🚫
헤더 없음
요청/응답 헤더가 없어 메타데이터 전달이 불가능했습니다. HTML 파일만 전송할 수 있었습니다.
🔌
연결 즉시 종료
응답 후 TCP 연결이 바로 닫힙니다. 요청마다 새 연결이 필요했습니다.
# 요청 (전부입니다) GET /index.html # 응답 (HTML 본문만, 헤더 없음) <html> <body>Hello, World!</body> </html>
HTTP/1.0
헤더와 상태 코드의 등장
1996 · RFC 1945
Header 추가 Status Code POST / HEAD 단발성 연결
▼

헤더(Header), 상태 코드, Content-Type이 도입되며 웹이 급속도로 발전했습니다. 하지만 치명적인 문제가 있었습니다. 요청마다 새로운 TCP 연결을 맺고 끊었기 때문에, 이미지 하나하나마다 3-way handshake가 발생해 페이지 로딩이 매우 느렸습니다.

📋
헤더 도입
Content-Type, Content-Length 등 메타데이터 전달이 가능해졌습니다. HTML 외 이미지, 파일 전송도 지원됩니다.
🔢
상태 코드
200 OK, 404 Not Found 등 응답 상태를 숫자 코드로 표현. 에러 처리가 가능해졌습니다.
📮
POST, HEAD 추가
데이터 전송(POST)과 헤더만 요청(HEAD)이 가능해지며 더 다양한 웹 애플리케이션 개발이 가능해졌습니다.
🔴
문제: 단발성 연결
요청마다 TCP 연결을 새로 수립합니다. 리소스 10개 로딩 시 TCP handshake가 10번 발생합니다.

HTTP/1.1
지속 연결과 파이프라이닝
1997 · RFC 2068 / 2616 · 현재 가장 많이 사용
Keep-Alive Pipelining Host 헤더 필수 캐시 제어 HOL Blocking
▼

현재도 가장 널리 사용되는 버전. Keep-Alive(지속 연결)로 TCP 연결을 재사용하고, Host 헤더를 통해 하나의 IP로 여러 도메인을 서비스(가상 호스팅)할 수 있게 됩니다. 파이프라이닝으로 응답을 기다리지 않고 요청을 연속 전송할 수 있게 됐지만, HOL Blocking(Head-of-Line Blocking) 문제가 발목을 잡습니다.

🔗
Keep-Alive (지속 연결)
TCP 연결을 유지하여 여러 요청에 재사용합니다. Connection: keep-alive 헤더로 제어합니다.
🏠
Host 헤더 필수화
하나의 IP에서 여러 도메인 서비스가 가능해졌습니다. 가상 호스팅(Virtual Hosting)의 기반입니다.
💾
향상된 캐시 제어
ETag, Cache-Control, If-None-Match 등 세밀한 캐시 정책이 가능해졌습니다. 304 Not Modified로 대역폭을 절약합니다.
🚧
HOL Blocking
앞선 요청이 지연되면 뒤의 모든 요청도 대기합니다. 파이프라이닝이 실질적으로 무력화되는 주요 원인입니다.

HTTP/2
멀티플렉싱과 바이너리 프레임
2015 · RFC 7540 · Google SPDY 기반
Multiplexing Binary Frame HPACK Server Push TCP HOL 잔존
▼

Google의 SPDY 프로토콜을 기반으로 HTTP의 대규모 개편이 이루어졌습니다. 멀티플렉싱으로 하나의 TCP 연결에서 여러 요청을 동시에 처리하고, 텍스트 대신 바이너리 프레임을 사용해 파싱 효율을 높였습니다. HPACK으로 헤더를 압축하고 서버 푸시도 지원합니다. 단, TCP 레벨의 HOL Blocking은 여전히 존재합니다.

🔀
멀티플렉싱 (Multiplexing)
하나의 TCP 연결 위에서 여러 스트림을 동시에 처리합니다. 요청·응답이 서로를 블로킹하지 않습니다.
📦
바이너리 프레임 (Binary)
텍스트 대신 이진(Binary) 형식으로 데이터를 전송합니다. 파싱이 빠르고 오류가 적습니다.
🗜️
HPACK 헤더 압축
중복 헤더를 압축해 전송합니다. 매번 반복 전송되던 Cookie, User-Agent 등이 대폭 줄어듭니다.
🚀
서버 푸시 (Server Push)
클라이언트가 요청하기 전에 서버가 필요한 리소스를 먼저 전송합니다. HTML 요청 시 CSS/JS를 미리 푸시할 수 있습니다.

HTTP/3
QUIC 기반의 차세대 HTTP
2022 · RFC 9114 · UDP + QUIC
QUIC UDP 기반 0-RTT HOL 완전 해결 Connection Migration
▼

TCP를 완전히 버리고 UDP 기반의 QUIC 프로토콜 위에서 동작합니다. HTTP/2의 TCP HOL Blocking을 근본적으로 해결하고, TLS 1.3을 내장해 연결 수립을 획기적으로 단축합니다. Wi-Fi ↔ LTE 전환 시에도 연결이 끊기지 않는 Connection Migration까지 지원합니다.

✅
HOL Blocking 완전 해결
스트림이 UDP 레벨에서 완전히 독립적입니다. 하나의 스트림 패킷 손실이 다른 스트림에 영향을 주지 않습니다.
⚡
0-RTT / 1-RTT 연결
최초 연결 1 RTT, 재연결 0 RTT. TCP+TLS의 2~3 RTT 대비 획기적으로 빠릅니다.
📱
Connection Migration
IP가 바뀌어도(Wi-Fi → LTE) Connection ID로 연결을 유지합니다. 모바일 환경에서 특히 유리합니다.
🔒
TLS 1.3 내장
보안이 프로토콜에 통합되어 있어 별도 TLS Handshake 단계가 필요 없습니다. QUIC = Transport + Security.

🎯 면접 핵심 포인트
🚧
HOL Blocking이란?
HTTP/1.1에서 앞선 요청의 응답 지연이 뒤의 모든 요청을 블로킹하는 현상. HTTP/2는 멀티플렉싱으로 Application 레벨을 해결했지만, TCP 패킷 손실 시 발생하는 TCP 레벨 HOL은 HTTP/3(QUIC)에서 완전 해결됩니다.
🔀
멀티플렉싱이란?
하나의 TCP 연결에서 여러 요청/응답을 동시에 처리하는 기법. 각 메시지를 프레임(Frame)으로 분할하고 스트림(Stream) ID를 붙여 전송한 뒤 재조립합니다. HTTP/2에서 처음 도입되었습니다.
⚡
HTTP/3는 왜 UDP를?
TCP의 HOL Blocking은 OS 커널 레벨에 고정된 구조라 해결 불가능합니다. UDP는 이 제약이 없어 QUIC이 스트림별 독립 재전송을 직접 구현할 수 있습니다. UDP 위에 신뢰성을 직접 구현한 것이 QUIC입니다.
🗜️
HPACK vs QPACK
HTTP/2의 HPACK은 동적 테이블을 공유해 헤더를 압축합니다. HTTP/3의 QPACK은 QUIC의 스트림 독립성을 고려해 테이블 동기화 방식을 개선한 버전입니다.
