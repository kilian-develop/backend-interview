DNS 동작 원리

브라우저에 URL을 입력했을 때, 도메인 이름이 IP 주소로 변환되기까지 일어나는 모든 과정 — 계층적 조회, 캐싱, 레코드 타입까지

DNS란 무엇인가?

DNS (Domain Name System)는 사람이 읽을 수 있는 도메인 이름을 컴퓨터가 이해할 수 있는 IP 주소로 변환하는 인터넷의 전화번호부입니다.

www.google.com 같은 이름을 기억하고 싶지만, 실제로 서버에 연결하려면 142.250.196.132 같은 IP가 필요합니다.

DNS는 이 변환을 계층적으로 분산된 데이터베이스를 통해 처리합니다.

# DNS 없이는... curl 142.250.196.132 # DNS 덕분에... curl https://www.google.com # DNS 조회 직접 해보기 nslookup www.google.com Non-authoritative answer: Name: www.google.com Address: 142.250.196.132 dig www.google.com +short 142.250.196.132
DNS 계층 구조 — 4단계 서버
💻
DNS Resolver
ISP 또는 공개 DNS 서버. 클라이언트 대신 전체 조회 과정을 수행합니다. Recursive Resolver라고도 합니다.
8.8.8.8 (Google)
1.1.1.1 (Cloudflare)
🌍
Root Name Server
DNS 계층의 최상위. TLD 서버의 주소를 알고 있습니다. 전 세계에 13개 루트 서버 클러스터(A~M)가 있습니다.
a.root-servers.net
198.41.0.4
🗂️
TLD Name Server
최상위 도메인(.com, .net, .kr 등)을 관리합니다. 해당 TLD 내 Authoritative 서버 주소를 알고 있습니다.
.com TLD 서버
.kr TLD 서버
📋
Authoritative NS
실제 도메인의 DNS 레코드를 보유한 최종 권한 서버입니다. 여기서 IP 주소를 최종 응답합니다.
ns1.google.com
ns2.google.com
🌍 루트 서버는 전 세계에 13개? — 정확히는 A~M 라벨의 13개 클러스터입니다. Anycast 기술로 각 클러스터가 전 세계 수백 개 서버로 분산되어 있습니다. 실제 물리적 서버는 1,000개 이상입니다.

DNS 조회 단계가 여기에 표시됩니다
DNS 레코드 타입
A
IPv4 주소 매핑
도메인을 IPv4 주소로 변환합니다. 가장 기본적이고 많이 사용되는 레코드입니다.
example.com. IN A 93.184.216.34
AAAA
IPv6 주소 매핑
도메인을 IPv6 주소로 변환합니다. 128비트 주소를 사용합니다.
example.com. IN AAAA 2606:2800::1
CNAME
별칭 (Alias)
다른 도메인의 별칭을 만듭니다. www를 apex 도메인으로 연결하는 데 주로 사용됩니다.
www.example.com. IN CNAME example.com.
NS
네임서버
도메인의 권한 있는 네임서버를 지정합니다. 도메인 등록 시 필수로 설정됩니다.
example.com. IN NS ns1.example.com.
MX
메일 서버
이메일을 받을 메일 서버를 지정합니다. 우선순위(Priority) 값이 낮을수록 먼저 시도됩니다.
example.com. IN MX 10 mail.example.com.
TXT
텍스트 데이터
임의의 텍스트를 저장합니다. 도메인 소유권 확인, SPF, DKIM 등 이메일 인증에 사용됩니다.
example.com. IN TXT "v=spf1 include:..."
PTR
역방향 조회
IP 주소에서 도메인 이름을 찾습니다. 역방향 DNS(rDNS) 조회에 사용됩니다.
34.216.184.93.in-addr.arpa PTR example.com.
SOA
권한 시작 (Start of Authority)
존(Zone)의 권한 정보와 타이머 값을 포함합니다. 각 DNS 존에 반드시 하나 있습니다.
시리얼번호, Refresh, Retry, Expire, TTL 포함
캐싱과 TTL (Time To Live)

DNS 응답은 TTL 값만큼 캐싱됩니다. TTL이 만료되면 다시 조회합니다. 캐시 히트 시에는 Root→TLD→Auth 서버에 조회 없이 바로 응답합니다.

브라우저 캐시TTL: ~60초
가장 짧음. 브라우저가 자체적으로 저장
OS / Local 캐시TTL: ~300초
/etc/hosts, nscd, systemd-resolved
Recursive ResolverTTL: ~3600초
ISP 또는 공개 DNS 서버 레벨
TLD 서버 캐시TTL: ~172800초
약 48시간. 변경 시 전파에 시간 소요
# DNS 조회 경로 (캐시 확인 순서) 1. 브라우저 DNS 캐시 chrome://net-internals/#dns 2. OS 캐시 (hosts 파일 포함) /etc/hosts 127.0.0.1 localhost 192.168.1.1 myserver.local 3. Recursive Resolver 캐시 (8.8.8.8, 1.1.1.1) 4. Root → TLD → Auth NS 순서 조회
💡 TTL 설정 전략
• 낮은 TTL (60초): 빠른 변경 가능 → 서버 마이그레이션, A/B 테스트
• 높은 TTL (86400초): DNS 트래픽 감소 → 안정적인 서비스
• 변경 전 TTL을 미리 낮게 설정하는 것이 좋은 관행입니다
재귀적 조회 vs 반복적 조회
🔁 재귀적 조회 (Recursive Query)
클라이언트가 Recursive Resolver에게 최종 답을 달라고 요청합니다. Resolver가 Root→TLD→Auth 모든 조회를 대신 수행해서 IP를 반환합니다.

클라이언트 ↔ Resolver 구간에서 사용됩니다.
# 클라이언트 → Resolver Q: www.example.com의 IP가 뭐야? A: 93.184.216.34 (최종 답 반환) # Resolver가 모두 대신 처리
🔄 반복적 조회 (Iterative Query)
Resolver가 각 DNS 서버에게 "다음에 물어볼 서버"를 응답받아 단계적으로 조회합니다. 서버가 최종 답 대신 참조할 서버 주소를 알려줍니다.

Resolver ↔ Root/TLD/Auth NS 구간에서 사용됩니다.
# Resolver → Root NS Q: www.example.com? A: 모름, .com TLD 서버 물어봐 # Resolver → TLD NS Q: www.example.com? A: 모름, ns1.example.com 물어봐

💡 DoH (DNS over HTTPS) / DoT (DNS over TLS): 기존 DNS는 평문으로 쿼리가 전송되어 ISP나 중간자가 조회 내용을 볼 수 있습니다. DoH/DoT는 DNS 쿼리를 암호화하여 프라이버시를 보호합니다.
🎯 면접 핵심 포인트
🔢
루트 서버는 왜 13개인가?
DNS 메시지는 UDP 512바이트 제한이 있었고, 모든 루트 서버 IP를 한 패킷에 담으려면 13개가 최대였습니다. 현재는 Anycast로 각 라벨당 수백 개 서버가 분산 운영됩니다.
⚡
DNS 캐시 포이즈닝
공격자가 Resolver의 캐시에 가짜 DNS 응답을 주입하는 공격입니다. DNSSEC(DNS Security Extensions)로 응답에 디지털 서명을 추가해 방어합니다.
🔄
CNAME vs A 레코드
A 레코드는 IP를 직접 반환하고, CNAME은 다른 도메인을 가리킵니다. CNAME은 추가 DNS 조회가 필요하고, apex 도메인(example.com)에는 CNAME을 사용할 수 없습니다.
📡
DNS는 UDP를 쓰는 이유?
DNS 쿼리는 작고 빠른 응답이 필요하기 때문에 연결 설정이 없는 UDP(포트 53)를 사용합니다. 응답이 512바이트를 초과하거나 Zone Transfer 시에는 TCP를 사용합니다.
🌐
TTL과 DNS 전파
DNS 변경 후 전파 시간은 기존 TTL 값에 따라 결정됩니다. 변경 전 TTL을 낮게(300초) 설정하고, 변경 완료 후 다시 높이는 것이 권장 관행입니다.
🔒
DoH vs DoT
DoH는 443 포트 HTTPS로 DNS를 전송 (방화벽 우회 용이). DoT는 853 포트 TLS 전용 (구분이 명확). 둘 다 DNS 쿼리를 암호화하여 프라이버시를 보호합니다.
